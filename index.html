Architecture

How Phonon works

A coordinator orchestrates a fleet of Android phones. Each phone runs a sidecar service that loads models, accepts inference requests, and reports health — all over your local network.

1 The Coordinator

A single Go binary runs on any device on your network — a Raspberry Pi, an Odroid, a spare laptop, or even a cloud VM. It serves an OpenAI-compatible API, discovers phones via mDNS, load-balances requests across available devices, and exposes a web UI for management.

$ phonon coordinator --config cluster.yaml
# Coordinator starts, begins mDNS discovery on 192.168.1.0/24
  • OpenAI-compatible API (/v1/chat/completions)
  • mDNS service discovery — phones announce themselves automatically
  • Round-robin load balancing across phones
  • Health monitoring with Prometheus metrics
  • YAML-based declarative configuration
  • Embedded React web UI

2 The Phone Sidecar

Each phone runs a Kotlin foreground service that connects to the coordinator, downloads models, and runs inference using LiteRT-LM — Google's NPU-first inference runtime. The sidecar reports battery level, temperature, and processing state every heartbeat.

  • Runs as a persistent foreground service with a notification
  • Inference via LiteRT-LM — NPU-accelerated on Pixel Tensor and Qualcomm Hexagon
  • Heartbeat and health reporting every 30 seconds
  • Model download, cache, and management
  • Battery-aware: reduces load when unplugged and below 20%
  • Pre-flight readiness checks before joining the pool

3 Three Deployment Modes

Pool Mode (Phase 1) — Available Now

Each phone runs independently. The coordinator load-balances requests round-robin. All phones work in parallel. Best for fast, high-throughput inference with smaller models (Gemma 4 E2B, Gemma 4 E4B).

Shard Mode (Phase 2+) — Coming

Multiple phones collectively run one large model via pipelined-ring parallelism. The coordinator routes to the group as a single logical device. Enables running models larger than any single phone can handle (Gemma 4 12B across 3-4 phones).

Public Pool — Coming

Donated or subscribed phones aggregated into a shared cluster at phononai.org. Authentik handles multi-tenant auth, Stripe handles billing, donated phones get priority scheduling.

4 Data Flow

A request to the Phonon coordinator looks like any other OpenAI API call. The flow is transparent to your application:

# Your app never knows it's talking to phones
curl https://phonon.local/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model":"gemma-4-e2b","messages":[{"role":"user","content":"Hello"}]}'
  1. Coordinator receives the request, selects an available phone
  2. Phone loads the model (cached from previous use) and runs inference via LiteRT-LM
  3. Tokens stream back through the coordinator to your client
  4. Phone reports updated health metrics (battery, temp, load)
  5. Coordinator logs the event and metrics to the event log

5 Visual Identity (Phase 2)

Each phone becomes a telemetry display. An animation framework on the sidecar renders inference state — token rate, temperature, battery, load — as real-time visualizations. Theme packs define the aesthetic (solarpunk, cyber HUD, synthwave, honeycomb, and more).

The visualization engine is workload-aware (state, token rate, temperature, battery) and not content-aware — it never sees prompts or responses.

  • Multi-phone coordinated animations (inference wave, temperature heatmap)
  • Community theme pack system via PRs
  • Web UI rack layout editor for physical placement mapping

Overview

Phonon turns old Android phones into a private AI inference cluster. A single Go coordinator binary runs on any Linux device on your network (Raspberry Pi, Odroid, spare laptop, cloud VM). One or more Android phones with the Phonon sidecar APK connect to the coordinator and form an inference pool.

The coordinator serves an OpenAI-compatible API. Any tool that can call /v1/chat/completions can use Phonon — no special integration needed.

Hardware Requirements

Coordinator

  • Any Linux device (Raspberry Pi 4+, Odroid, x86 machine)
  • 1 GB RAM minimum, 2 GB+ recommended
  • Network access to your phone fleet (same LAN or Tailscale)
  • Docker or Go runtime

Phones

  • Android 14+
  • NPU recommended (Pixel 6+, Galaxy S24+, OnePlus 12+ — any device with a Tensor or Hexagon NPU)
  • 4 GB+ RAM (8 GB+ for larger models like Gemma 4 E4B)
  • Available via ADB or network pairing

Installation

1. Install the coordinator

docker pull chezgoulet/phonon:latest
docker run -d \
  --name phonon \
  -p 8080:8080 \
  -v /path/to/config.yaml:/etc/phonon/config.yaml \
  chezgoulet/phonon:latest

Or run the binary directly:

# Download the latest release from GitHub
wget https://github.com/chezgoulet/phonon/releases/latest/download/phonon-linux-arm64
chmod +x phonon-linux-arm64
./phonon-linux-arm64 coordinator --config config.yaml

2. Install the APK on each phone

Recommended (Obtainium): Add the GitHub releases URL to Obtainium for automatic updates. The APK is release-signed and compatible with sideloading.

Manual: Download the APK from the latest GitHub Release and sideload via ADB:

adb install phonon-sidecar-release.apk

3. Pair phones to the coordinator

Open the Phonon sidecar app. It discovers the coordinator via mDNS automatically. Accept the pairing prompt. The phone is now part of your cluster.

Configuration

The coordinator uses a YAML configuration file. Here's a minimal example:

cluster:
  name: "home-pool"
  listen: ":8080"

discovery:
  mdns: true
  mdns_interface: "eth0"

models:
  - name: "gemma-4-e2b"
    url: "https://huggingface.co/google/gemma-4-e2b-litert"
    sha256: "abc123..."
    min_ram_mb: 4096

telemetry:
  prometheus: true
  event_log: "sqlite:///var/lib/phonon/events.db"

Usage

Once your cluster is running, point any OpenAI-compatible client at the coordinator:

curl http://phonon.local:8080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gemma-4-e2b",
    "messages": [{"role": "user", "content": "Hello!"}]
  }'

Theme Packs

Phonon's sidecar includes a visualization engine that renders phone state (inference load, temperature, battery) as animated displays. Theme packs are community-contributed Kotlin files that implement the VisualizationPack interface.

Built-in themes: Cyber HUD, Neon Ring, Matrix Rain, Bioluminescent, LCARS, Synthwave, Honeycomb, Veil.

Submit your own theme via a PR on the phonon repo. Instructions are in CONTRIBUTING.md.

Troubleshooting

Phone not discovered

Check that the phone and coordinator are on the same network. Verify mDNS is enabled on your router. Try restarting the sidecar app.

Model download fails

Ensure the phone has internet access for the initial download. Models are cached after download. Check available storage space.

Poor inference performance

Verify the phone has an NPU and that LiteRT-LM is using it. Check battery level — phones below 20% and not charging enter degraded mode. Check temperature — thermal throttling reduces performance.

About

Why Phonon exists

AI inference is becoming a utility — it should be as accessible, private, and sustainable as the devices we already own.

🌱 The problem

AI inference is expensive, centralized, and generates e-waste. Cloud providers charge per-token at margins that reflect datacenter costs, not compute value. Their hardware turns over every 2-3 years. And every upgrade cycle creates more specialized waste.

Meanwhile, the phone in your pocket — or the one in your drawer — has an NPU capable of running modern language models at usable speeds. It's already paid for. It already exists. It's already connected to power and network.

💡 The insight

Phonon started with a simple observation: a Pixel 7a's Tensor NPU can run a Gemma 4 E2B model at ~55 tokens/second. That's not fast enough to replace a GPU for training, but it's more than fast enough for agent inference, batch processing, and personal AI workloads.

Ten phones in a drawer become a ~500 tok/s cluster. For free. With zero e-waste, zero cloud dependency, and total data privacy.

🔧 The architecture

Phonon is two components: a Go coordinator and an Android sidecar APK. The coordinator serves an OpenAI-compatible API and load-balances across phones. The sidecar runs inference via LiteRT-LM, Google's NPU-first runtime. Everything communicates over your local network. No telemetry. No phone-home. No cloud required.

👤 The team

Phonon is built by a single human and two AI agents — Vigilant (system architect, devops) and Dev (implementation). A one-person operation amplified by agents, running on a shoestring budget with unfashionable hardware.

Based in Burlington, Vermont. Moving to Sherbrooke, Quebec. The server runs on a Linode VPS in Toronto and an Odroid at home. The phones come from eBay and Swappa. Total budget for the initial build: ~$50.

⚖️ License

Phonon is Apache 2.0 — fully permissive. The core coordinator, sidecar, and all built-in theme packs are open source. The public pool and hosted services are complementary offerings, not gated features. Self-hosted is always fully functional.

Quickstart

Get your cluster running in 15 minutes

One coordinator, a few phones, and you'll have private inference running on your local network.

Set up the coordinator

You need a Linux device on your network. A Raspberry Pi 4+, an Odroid N2+, or any x86 machine with Docker. Run:

docker pull chezgoulet/phonon:latest
mkdir -p ~/phonon/config
docker run -d --name phonon \
  -p 8080:8080 \
  -v ~/phonon/config:/etc/phonon \
  chezgoulet/phonon:latest

Install the APK on phones

On each phone you want in the cluster, install the Phonon sidecar APK. The easiest way is via Obtainium — add the GitHub releases URL and it'll auto-update.

Or sideload manually:

# Download from latest GitHub Release
wget https://github.com/chezgoulet/phonon/releases/latest/download/phonon-sidecar-release.apk
adb install phonon-sidecar-release.apk

Pair phones

Open the Phonon app on each phone. It discovers the coordinator automatically via mDNS. Accept the pairing prompt. The phone joins the pool.

If mDNS doesn't work on your network, enter the coordinator's IP address manually in the app's settings. Both devices must be on the same LAN (or connected via Tailscale).

Download a model

Models are downloaded on-demand when a request comes in. The coordinator manages model distribution. Make your first request:

curl http://phonon.local:8080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model": "gemma-4-e2b", "messages": [{"role": "user", "content": "Hello!"}]}'

Point your agents at it

Configure any OpenAI-compatible tool to use your Phonon endpoint:

# Hermes Agent
model_provider: custom
custom:
  base_url: http://phonon.local:8080
  api_key: phonon-local

# Or any OpenAI client
openai.base_url = "http://phonon.local:8080"

Check the dashboard

Open the coordinator's web UI in your browser to see your cluster's health, token rates, temperature, and battery levels for every phone.

open http://phonon.local:8080
